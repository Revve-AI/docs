A **Scorecard** (**Evaluations → Scorecards** in the left dashboard nav, visible with the **Manage Agents** permission) is a reusable scoring policy: a named bundle of metrics from the shared [Metrics library](/evaluations/evaluation-metrics), each with a weight, a single speaker scope (Agent or Caller), and a minimum passing score. The same scorecard grades calls in two different ways depending on how you use it — attach it to a Voice Agent's **Evaluation** tab and it grades every live call that agent completes, automatically; or upload a batch of recordings here and it grades every call in that batch. Same policy, same metrics, two different triggers — you don't build the grading logic twice.

[Evaluators and Runs](/evaluations/creating-an-evaluator) grade *simulated* conversations before you ship a change — pre-ship regression testing against a persona or fixed script. A Scorecard grades conversations that already happened, in one of two ways: automatically, right after a live call completes, for every Voice Agent it's attached to; or in a batch, for recordings you upload after the fact — a QA sample of last month's collections calls, recordings exported from a legacy dialer before you migrate, or any call made outside Revve entirely.

## Where Scorecards fits

Everything that grades a conversation in Revve draws its metrics from the same library:

| System | Grades | When |
|--------|--------|------|
| [Metrics library](/evaluations/evaluation-metrics) | Nothing itself — it's the shared library of judging criteria the other three attach. | — |
| **Scorecards**, attached to a Voice Agent | Every live production call, for that agent — via the [Evaluation tab](/voice-agents/voice-agent-evaluation), which is just where you pick which of your team's scorecards apply to this agent. | Automatically, after each call completes. |
| **Scorecards**, run as an Upload Batch | Uploaded recordings of real calls, in batches. | Whenever you upload a batch — QA sampling, compliance audits, historical or external calls. |
| [Evaluators / Runs](/evaluations/creating-an-evaluator) | Simulated or scripted test conversations. | Before you publish a change — pre-ship regression testing. |

Because a scorecard is just a policy over shared metrics, editing a metric's definition in the library changes how every scorecard that uses it grades — the scorecard only owns the metric's weight within its policy.

## The Scorecards page

The page ("Reusable scoring policies for evaluating uploaded call recordings") lists your team's scorecards in a table:

| Column | Meaning |
|--------|---------|
| **Name** | The scorecard's name. Click the row to open it for editing. |
| **Metrics** | How many metrics the policy includes. |
| **Updated** | When the policy was last changed. |

Before you've created any, the empty state reads "No scorecards yet — Create your first scorecard to start scoring uploaded recordings."

## Creating and editing a scorecard

Click **New scorecard** (top right) to open a sheet from the right — "Pick metrics and weights to score completed call recordings." Clicking an existing row opens the same sheet as **Edit scorecard**.

| Field | Notes |
|-------|-------|
| **Name** | Required. Placeholder: "e.g. Compliance scorecard". Name it after the *question the audit answers* (Compliance, Greeting quality, Collections script adherence), not after the batch you're about to upload — the whole point is reusing it across batches. |
| **Description** | Placeholder: "What does this scorecard measure?". Worth filling in — it's what tells a teammate whether to reuse your scorecard or create another. |
| **What do you want to evaluate?** | Required, radio: **Agent** ("Score how the AI agent handled the call") or **Caller** ("Score the caller's side of the conversation"). Locks the scorecard to one speaker for its lifetime, with helper text below confirming it — for example, `Only Agent metrics can be added below` — and the metric picker only offers metrics built for that scope. If you need both sides graded, build two scorecards, one of each scope, and attach both wherever you need live grading. |
| **Judge Model** | Which LLM grades this scorecard's LLM Judge metrics — provider, model, and reasoning effort. Defaults to **openai** / **o4-mini**. Change it when a metric needs a stronger (or cheaper) judge than the default; every LLM Judge metric on this scorecard shares the same judge model, so split judgment-heavy and cheap-check-heavy metrics across separate scorecards if you want different judges for each. |
| Weighted metrics | The policy itself: pick metrics from the shared [Metrics library](/evaluations/evaluation-metrics) (filtered to the scope you chose above), give each a weight, and set a minimum passing score (1–5). See [Voice Agent Evaluation — Scoring math](/voice-agents/voice-agent-evaluation#scoring-math) for how weights and the minimum passing score combine into a pass/fail call. |

Submit with **Create scorecard** (new) or **Save changes** (editing).

<Tip>
Weight by what a failure costs you, not by how easy the metric is to check. In a compliance scorecard, "read the required disclosure" should be able to sink the score on its own — give it a dominant weight — while style metrics like tone share the remainder. A flat equal-weights policy lets three polite pleasantries outvote one missed disclosure.
</Tip>

## Using a scorecard for live grading

A scorecard on its own doesn't grade anything — it's a policy waiting to be pointed at calls. To grade a Voice Agent's live calls automatically, attach the scorecard from that agent's **Evaluation** tab; every call the agent completes from then on is scored against it. See [Voice Agent Evaluation](/voice-agents/voice-agent-evaluation) for the attach/detach UI and how the attached scorecards' scores show up in Call History.

The rest of this page — Upload Batches, below — covers the other trigger: grading recordings you upload after the fact, rather than live calls as they complete.

## Upload Batches

Once you've uploaded at least one batch of recordings, an **Upload Batches** section appears below the scorecards table — "Batches of uploaded call recordings queued for scoring" Scoring is asynchronous: a batch queues, processes each file, and reports progress here.

| Column | Meaning |
|--------|---------|
| **Batch** | The batch's name. Click it to open **Call History** filtered to just that batch's calls. |
| **Status** | `completed`, `processing`, `failed`, or `partial` (some files scored, some didn't). |
| **Progress** | Processed count out of total files. |
| **Failed** | How many files couldn't be processed. |
| **Created** | When the batch was uploaded. |

<Note>
The Scorecards page defines the policy and tracks batch progress — that's all. Uploading the recordings themselves, and reviewing each scored call afterward, both happen on the **Call History** side. Clicking a batch name here is the shortcut across: it lands you in Call History pre-filtered to that batch.
</Note>

A `partial` or `failed` status means some files never got scored — check the **Failed** count before treating the batch's results as a complete sample, especially for a compliance audit where the unscored calls are exactly the ones you can't vouch for.

## When to reach for a scorecard

| Situation | Why a scorecard |
|-----------|-----------------|
| Compliance audit of a sample of last month's collections calls | One weighted policy applied identically to every sampled recording — auditable, repeatable month over month. |
| Scoring recordings from a legacy dialer before migrating to Revve | The calls never touched a Revve agent, so no Evaluation tab ever graded them; upload and grade them against the same metrics your Revve agents use, for a like-for-like baseline. |
| Spot-checking agent quality with a weighted rubric | Cheaper than grading every live call — sample, upload, score, repeat with the same scorecard so trends are comparable. |
