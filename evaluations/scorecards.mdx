A **Scorecard** (**Evaluations → Scorecards** in the left dashboard nav, visible with the **Manage Agents** permission) is a reusable scoring policy for one side of a conversation — a named bundle of metrics from the shared [Metrics library](/evaluations/evaluation-metrics), each with a weight, scoped to either the **Agent** or the **Caller**, plus a single minimum passing score. Once built, a scorecard scores real calls the same way every time, so "did this call pass?" means the same thing across a hundred recordings and across months of audits.

This page covers building a scorecard and applying it to a batch of **recordings you upload** — a QA sample of last month's collections calls, recordings exported from a legacy dialer before you migrate, or any call made outside Revve entirely. A scorecard isn't limited to batches, though: the same object can also be attached to a voice agent's [Evaluation tab](/voice-agents/voice-agent-evaluation) to grade that agent's live calls automatically, call after call — that attach/detach flow is covered there, not here.

## Where Scorecards fits

Three things in Revve grade conversations, and they all draw metrics from the same library:

| System | Grades | When |
|--------|--------|------|
| [Metrics library](/evaluations/evaluation-metrics) | Nothing itself — it's the shared library of judging criteria the other two attach. | — |
| [Evaluators / Runs](/evaluations/creating-an-evaluator) | Simulated or scripted test conversations. | Before you publish a change — pre-ship regression testing. |
| **Scorecards** | Real calls — either every live call for an agent it's attached to, or every recording in a batch you upload. | Automatically after each live call (attached path), or whenever you upload a batch (this page's path). |

A scorecard is the one reusable policy object in Revve: build it once here, then either attach it to a voice agent for ongoing live-call grading or apply it directly to a batch of recordings you already have — the same weights, the same minimum passing score, the same metrics either way. Editing a metric's definition in the library changes how every scorecard that uses it grades, wherever that scorecard is consumed — a scorecard only owns the metric's weight within its own policy.

## The Scorecards page

The page ("Reusable scoring policies for evaluating uploaded call recordings") lists your team's scorecards in a table:

| Column | Meaning |
|--------|---------|
| **Name** | The scorecard's name. Click the row to open it for editing. |
| **Metrics** | How many metrics the policy includes. |
| **Updated** | When the policy was last changed. |

Before you've created any, the empty state reads "No scorecards yet — Create your first scorecard to start scoring uploaded recordings."

## Creating and editing a scorecard

Click **New scorecard** (top right) to open a sheet from the right — "Pick metrics and weights to score completed call recordings." Clicking an existing row opens the same sheet as **Edit scorecard**.

| Field | Notes |
|-------|-------|
| **Name** | Required. Placeholder: "e.g. Compliance scorecard". Name it after the *question the audit answers* (Compliance, Greeting quality, Collections script adherence), not after the batch you're about to upload — the whole point is reusing it across batches. |
| **Description** | Placeholder: "What does this scorecard measure?". Worth filling in — it's what tells a teammate whether to reuse your scorecard or create another. |
| **What do you want to evaluate?** | **Agent** ("Score how the AI agent handled the call") or **Caller** ("Score the caller's side of the conversation"). Pick one — a scorecard scores exactly one side, not both, and the choice can't be split later. Need both sides graded on the same calls? Build two scorecards, one per speaker, and apply (or attach) both. |
| Weighted metrics | The policy itself — see below. |
| **Judge Model** | Which LLM judges this scorecard's LLM-judged metrics (OpenAI or Anthropic). Leave unset to fall back to the platform default judge. |

Pick the speaker scope first: it filters the metric picker below to only that speaker's metrics, so add metrics after you've chosen Agent or Caller. The picker itself is the same weighted-metrics editor used across Evaluations — pick metrics from the shared [Metrics library](/evaluations/evaluation-metrics), give each a weight, and set one minimum passing score (1–5) for the whole scorecard. If you already know that editor from an agent's Evaluation tab, there's nothing new to learn beyond the speaker-scope and Judge Model fields above.

A metric can also be marked informational-only when you create it in the Metrics library — it's still judged and shown on every scored call, just excluded from the scorecard's overall pass/fail score. See [Evaluation Metrics](/evaluations/evaluation-metrics) for that flag.

Submit with **Create scorecard** (new) or **Save changes** (editing).

<Tip>
Weight by what a failure costs you, not by how easy the metric is to check. In a compliance scorecard, "read the required disclosure" should be able to sink the score on its own — give it a dominant weight — while style metrics like tone share the remainder. A flat equal-weights policy lets three polite pleasantries outvote one missed disclosure.
</Tip>

## Upload Batches

Once you've uploaded at least one batch of recordings, an **Upload Batches** section appears below the scorecards table — "Batches of uploaded call recordings queued for scoring" Scoring is asynchronous: a batch queues, processes each file, and reports progress here.

| Column | Meaning |
|--------|---------|
| **Batch** | The batch's name. Click it to open **Call History** filtered to just that batch's calls. |
| **Status** | `completed`, `processing`, `failed`, or `partial` (some files scored, some didn't). |
| **Progress** | Processed count out of total files. |
| **Failed** | How many files couldn't be processed. |
| **Created** | When the batch was uploaded. |

<Note>
The Scorecards page defines the policy and tracks batch progress — that's all. Uploading the recordings themselves, and reviewing each scored call afterward, both happen on the **Call History** side. Clicking a batch name here is the shortcut across: it lands you in Call History pre-filtered to that batch.
</Note>

A `partial` or `failed` status means some files never got scored — check the **Failed** count before treating the batch's results as a complete sample, especially for a compliance audit where the unscored calls are exactly the ones you can't vouch for.

## When to reach for a scorecard

| Situation | Why a scorecard |
|-----------|-----------------|
| Compliance audit of a sample of last month's collections calls | One weighted policy applied identically to every sampled recording — auditable, repeatable month over month. |
| Scoring recordings from a legacy dialer before migrating to Revve | The calls never touched a Revve agent, so no Evaluation tab ever graded them; upload and grade them against the same metrics your Revve agents use, for a like-for-like baseline. |
| Spot-checking agent quality with a weighted rubric | Cheaper than grading every live call — sample, upload, score, repeat with the same scorecard so trends are comparable. |
