A **Scorecard** (**Evaluations → Scorecards** in the left dashboard nav, visible with the **Manage Agents** permission) is a reusable scoring policy — a named bundle of metrics from the shared [Metrics library](/evaluations/evaluation-metrics), each with a weight, plus a minimum passing score, scoped to one side of the conversation (Agent or Caller). Upload a batch of recordings, and every call in the batch gets graded against the scorecard the same way, so "did this call pass?" means the same thing across a hundred recordings and across months of audits.

[Evaluators and Runs](/evaluations/creating-an-evaluator) grade *simulated* conversations before you ship a change; a Scorecard grades *real* calls — either a batch you upload after the fact (a QA sample of last month's collections calls, recordings from a legacy dialer, any call made outside Revve entirely), or every live call one of your voice agents completes going forward. A scorecard doesn't pick which of those it does — it's a reusable policy, and where you attach it decides: upload a batch here on the Scorecards page, or attach the scorecard to a voice agent for automatic grading of its live calls. See [Attaching a Scorecard to a Voice Agent](#attaching-a-scorecard-to-a-voice-agent) below.

## Where Scorecards fits

Three things in Revve grade conversations, and they all draw metrics from the same library:

| System | Grades | When |
|--------|--------|------|
| [Metrics library](/evaluations/evaluation-metrics) | Nothing itself — it's the shared library of judging criteria the other two attach. | — |
| [Evaluators / Runs](/evaluations/creating-an-evaluator) | Simulated or scripted test conversations. | Before you publish a change — pre-ship regression testing. |
| **Scorecards** | Uploaded recordings of real calls, in batches — *and* every live call of any voice agent the scorecard is attached to. | Whenever you upload a batch, or automatically after each live call once attached. |

A voice agent's own [Evaluation tab](/voice-agents/voice-agent-evaluation) is where you view and manage that agent's attached scorecards and their grading results — it doesn't define a separate grading mechanism of its own; it's a consumer of the same scorecards you author here.

Because a scorecard is just a policy over shared metrics, editing a metric's definition in the library changes how every scorecard that uses it grades — the scorecard only owns the metric's weight within its policy.

## Attaching a Scorecard to a Voice Agent

A scorecard isn't limited to one use. The same named scorecard can be attached to one or more voice agents — so every live call those agents complete is graded automatically — while still being available here to grade uploaded batches. Write the policy once (a compliance bar, a greeting-quality rubric) and reuse it everywhere that policy needs to apply, rather than maintaining near-duplicate scorecards per use case.

Attaching happens on the voice agent's own Evaluation tab, not on this page — an **Attach Scorecard** button there lists your team's scorecards to attach. Once attached, each graded live call shows that scorecard's result in **Call History** alongside the call. See [Voice Agent Evaluation](/voice-agents/voice-agent-evaluation) for the full attach workflow.

## The Scorecards page

The page ("Reusable scoring policies for evaluating uploaded call recordings") lists your team's scorecards in a table:

| Column | Meaning |
|--------|---------|
| **Name** | The scorecard's name. Click the row to open it for editing. |
| **Metrics** | How many metrics the policy includes. |
| **Updated** | When the policy was last changed. |

Before you've created any, the empty state reads "No scorecards yet — Create your first scorecard to start scoring uploaded recordings."

## Creating and editing a scorecard

Click **New scorecard** (top right) to open a sheet from the right — "Pick metrics and weights to score completed call recordings." Clicking an existing row opens the same sheet as **Edit scorecard**.

| Field | Notes |
|-------|-------|
| **Name** | Required. Placeholder: "e.g. Compliance scorecard". Name it after the *question the audit answers* (Compliance, Greeting quality, Collections script adherence), not after the batch you're about to upload — the whole point is reusing it across batches. |
| **Description** | Placeholder: "What does this scorecard measure?". Worth filling in — it's what tells a teammate whether to reuse your scorecard or create another. |
| Agent / Caller scope | Which side of the conversation this scorecard evaluates. Only metrics matching the chosen scope can be added to the policy. Wherever a scorecard's name appears — this list, a voice agent's attach dialog, Call History — a small Bot or User icon next to it shows the scope at a glance (tooltip: "Agent scorecard" or "Caller scorecard"). |
| Judge Model | Which LLM judges this scorecard's LLM Judge metrics — OpenAI or Anthropic, with a model choice underneath. Leave unset to fall back to the platform default judge model. This is separate from an agent's own LLM: it only affects how this scorecard's metrics are scored. |
| Weighted metrics | The policy itself — see below. |

The policy editor is built from the **same weighted-metrics components as the Voice Agent Evaluation tab**: pick metrics from the shared [Metrics library](/evaluations/evaluation-metrics), give each a weight, and set a minimum passing score (1–5). The metric picker only offers metrics matching the scope you chose above — an Agent scorecard can't accidentally pick up a Caller-only metric, and vice versa. If you already know that editor from an agent's Evaluation tab, there's nothing new to learn here; the scoring mechanics are documented in [Voice Agent Evaluation](/voice-agents/voice-agent-evaluation).

Submit with **Create scorecard** (new) or **Save changes** (editing).

A metric can be flagged **Informational only** in the [Metrics library](/evaluations/evaluation-metrics) — that's set on the metric's own definition, not here. A scorecard still judges an informational metric and shows its result, but excludes it from the scorecard's overall score, so you can track a metric without letting it move the pass/fail line.

<Tip>
Weight by what a failure costs you, not by how easy the metric is to check. In a compliance scorecard, "read the required disclosure" should be able to sink the score on its own — give it a dominant weight — while style metrics like tone share the remainder. A flat equal-weights policy lets three polite pleasantries outvote one missed disclosure.
</Tip>

## Upload Batches

Once you've uploaded at least one batch of recordings, an **Upload Batches** section appears below the scorecards table — "Batches of uploaded call recordings queued for scoring" Scoring is asynchronous: a batch queues, processes each file, and reports progress here.

| Column | Meaning |
|--------|---------|
| **Batch** | The batch's name. Click it to open **Call History** filtered to just that batch's calls. |
| **Status** | `completed`, `processing`, `failed`, or `partial` (some files scored, some didn't). |
| **Progress** | Processed count out of total files. |
| **Failed** | How many files couldn't be processed. |
| **Created** | When the batch was uploaded. |

<Note>
The Scorecards page defines the policy and tracks batch progress — that's all. Uploading the recordings themselves, and reviewing each scored call afterward, both happen on the **Call History** side. Clicking a batch name here is the shortcut across: it lands you in Call History pre-filtered to that batch.
</Note>

A `partial` or `failed` status means some files never got scored — check the **Failed** count before treating the batch's results as a complete sample, especially for a compliance audit where the unscored calls are exactly the ones you can't vouch for.

## When to reach for a scorecard

| Situation | Why a scorecard |
|-----------|-----------------|
| Compliance audit of a sample of last month's collections calls | One weighted policy applied identically to every sampled recording — auditable, repeatable month over month. |
| Scoring recordings from a legacy dialer before migrating to Revve | The calls never touched a Revve agent, so no Evaluation tab ever graded them; upload and grade them against the same metrics your Revve agents use, for a like-for-like baseline. |
| Spot-checking agent quality with a weighted rubric | Cheaper than grading every live call — sample, upload, score, repeat with the same scorecard so trends are comparable. |
