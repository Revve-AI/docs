# Automation Rules — Overview

![Automation Rules list showing rules with event types, status, and conditions](/screenshots/automation-rules-list.png)

**Automation Rules** let you trigger actions automatically when specific events happen in the system. Instead of manually updating CRM records after every call or sending notifications when an enrollment completes, you define rules that do it for you.

## How it works

Each automation rule has three parts:

1. **Event trigger** — what event fires the rule (e.g., a call was analyzed, a contact was updated).
2. **Conditions** — filters that determine whether this specific event should activate the rule (e.g., only for a certain agent, only when sentiment is negative).
3. **Actions** — what to do when the conditions are met (e.g., call a webhook, update a contact field, send an email, create a ticket).

When an event occurs, the system checks all active rules for that event type. If a rule's conditions match, its actions execute automatically.

## Available event types

| Event type | When it fires |
|------------|---------------|
| **Call Analyzed** | After a voice call is transcribed and analyzed (summary, sentiment, extracted fields are ready). |
| **Chat Analyzed** | After a chat conversation is analyzed. |
| **Node Moved** | When a contact transitions between nodes in a Conversational Flow. |
| **Lead Created** | When a new lead/contact is created in the system. |
| **Contact Updated** | When a contact record is modified. |
| **Enrollment Created** | When a lead is enrolled into a campaign. |
| **Enrollment Completed** | When a lead exits a campaign (success or failure). |
| **Enrollment Paused** | When an enrollment is paused (e.g., prehook failure). |
| **Action Completed** | When a campaign action (call, message) completes successfully. |
| **Action Failed** | When a campaign action fails. |

## Available actions

| Action | What it does |
|--------|-------------|
| **API Call (Webhook)** | Send an HTTP request to any endpoint. Supports GET, POST, PUT, PATCH, DELETE with custom headers, body, proxy, and timeout. |
| **Update Contact Field** | Set one or more fields on the contact record associated with the event. |
| **Send Email** | Send an email with configurable to/cc/bcc, subject, and HTML body. |
| **Create Ticket** | Create a support ticket with type, title, description, and custom fields. |

## Condition operators

Conditions use **dot-notation paths** into the event payload (e.g., `payload.contact.email`, `payload.call.sentiment`). Available operators:

| Operator | Meaning |
|----------|---------|
| `equals` / `not_equals` | Exact match / not match |
| `contains` / `not_contains` | Substring match |
| `starts_with` / `ends_with` | Prefix / suffix match |
| `gt` / `gte` / `lt` / `lte` | Numeric comparisons |
| `exists` / `not_exists` | Field is present / absent |
| `in` / `not_in` | Value is in / not in a list |

All conditions on a rule must match (AND logic). To create OR logic, use separate rules for each branch.

## Where to find Automation Rules

In the left sidebar, expand **Automations** and click **Automation Rules**. You will see:

- A searchable list of all rules
- Filters by **Event Type** and **Status** (Active / Inactive)
- Each rule shows its name, event type, status, creation date, creator, and the number of conditions and actions

## Execution history

To see what rules have fired and what they did, go to **Automations → Executions**. This log shows:

- Which rule fired
- When it fired
- The event that triggered it
- Whether it succeeded or failed
- Click any execution to see the full event payload and action results
