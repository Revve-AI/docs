title: "Answer Machine Detection"
description: "Classify who picked up — human, voicemail, or IVR menu — before your agent speaks, and decide what happens next: leave a message, navigate the menu, or hang up."
---

On outbound campaigns, a large share of dials never reach a person — they hit voicemail boxes and phone menus. **Answer Machine Detection** (AMD) classifies the callee as a **human**, **voicemail**, or **IVR menu** *before* the agent speaks, so your agent doesn't deliver its opening pitch to an answering machine. You configure it in the **Answer Machine Detection** tab of your agent's Advanced Settings (this tab was previously named "Voicemail").

The payoff is twofold: humans get a clean, immediate conversation start, and machines get exactly the handling you chose — a voicemail message, automated menu navigation, or a quick hangup that doesn't burn call minutes.

## How detection works

What happens on pickup depends on whether **Answer Machine Detection** is on:

1. **AMD on (native detection).** The moment the call connects, native detection listens to the greeting and classifies the callee before the agent says anything:
   - **Human** — the agent starts the conversation normally.
   - **Voicemail** — the agent leaves your voicemail message, or hangs up if **Leave Voicemail Message** is off.
   - **IVR menu** — the agent navigates the phone menu (DTMF prompts), or hangs up if **IVR Navigation** is off.
2. **AMD off (legacy fallback).** The agent falls back to LLM-based voicemail detection *during* the conversation — it starts speaking, then decides mid-call whether it's talking to a machine. This works, but the agent may talk over a voicemail greeting before the detection lands.

<Note>
AMD is off by default and opt-in per agent. Existing agents keep their current behavior until you enable it.
</Note>

## Enabling it

Open your agent's Advanced Settings, select the **Answer Machine Detection** tab, and switch on the **Answer Machine Detection** toggle at the top. Enabling it reveals the **IVR Navigation** card and the **Detection Options** thresholds below.

Turn it on for outbound agents — that's where machines answer most often, and classifying before the agent speaks gives every outcome a cleaner start. If you leave it off, the **Leave Voicemail Message** settings below still apply via the legacy LLM-based path.

## Leave Voicemail Message

The **Leave Voicemail Message** card controls what happens when voicemail is detected — on either detection path. If enabled, the agent leaves a voicemail message when voicemail is detected; if disabled, the call hangs up.

| Setting | Default | When to change |
|---------|---------|----------------|
| **Leave Voicemail Message** | Off | Turn on for outbound campaigns. A personalized voicemail measurably improves callback rates versus silent hangups. Keep off only when a message adds no value (e.g., pure verification calls). |
| **Voicemail Message** | Empty | Write it like a human would: who's calling, why, and how to call back. Supports contact placeholders — "Hi `{{firstName}}`, this is Acme calling about your application…". Keep it short; voicemail boxes cut long messages off. |
| **Detection Timeout (ms)** | 5000 (1000–30000) | Applies only to the legacy (LLM-based) detection path used when AMD is off — how long that fallback listens before deciding. Leave it unless legacy detection misfires. With AMD on, tune **Detection Options** instead. |

## IVR Navigation

The **IVR Navigation** card appears when AMD is on. When enabled, the agent automatically navigates IVR / phone-menu (DTMF) prompts — pressing through "press 1 for sales" style menus to reach a person. When disabled, the call hangs up if an IVR menu is detected.

| Setting | Default | When to change |
|---------|---------|----------------|
| **IVR Navigation** | Off | Turn on when your contact list includes business lines — clinics, offices, support desks — where numbers route through a phone menu before a human answers. Keep off for consumer lists, where an IVR usually means a wrong number. |

## Detection Options

The **Detection Options** section holds the advanced timing thresholds behind native detection. Leave them at the defaults unless you need to tune detection for specific carriers.

| Setting | Default | When to change |
|---------|---------|----------------|
| **Human speech threshold (s)** | 2.5 (0–30) | Seconds of continuous speech that classify the callee as a human. Long, chatty voicemail greetings getting classified as human → raise it. Talkative humans classified as machines → lower it. |
| **Human silence threshold (s)** | 0.5 | Post-speech silence treated as a human end-of-turn. Humans pause after "Hello?"; machines keep talking. Leave it unless support suggests otherwise. |
| **Machine silence threshold (s)** | 1.5 | Post-speech silence treated as a machine greeting boundary — the gap after the greeting where the beep lands. Voicemail messages starting too early or too late relative to the beep → adjust here. |
| **No-speech threshold (s)** | 10.0 (0–60) | Seconds with no speech before detection returns "uncertain". Lower it to fail faster on dead-air pickups; raise it for lines that connect slowly. |
| **Detection timeout (s)** | 20.0 (1–120) | The overall detection budget before a verdict is forced. Lower it to cap time spent classifying; raise it for carriers with long connection audio. |
| **Classification Prompt (optional)** | Blank | Overrides the prompt used to classify the greeting. Leave blank to use the built-in answering-machine classification prompt — override only if the built-in one consistently misclassifies your specific call patterns. |

## Troubleshooting detection

Work from the misclassification you observe, and change one threshold at a time:

- **Voicemail greetings classified as human.** Long personal greetings ("Hi, you've reached Alex, I can't come to the phone right now…") can exceed the human speech threshold. Raise **Human speech threshold (s)** so continuous machine-style speech no longer counts as human.
- **Humans classified as voicemail.** A person who answers with a long spiel can look like a machine. Lower **Human speech threshold (s)** slightly, or check Call History transcripts to see what the greeting actually sounded like.
- **Silent pickups hang in detection.** If callees pick up and say nothing, detection waits out **No-speech threshold (s)** before returning "uncertain". Lower it if these calls are common on your list.
- **Voicemail message clipped at the start.** The message may be starting before the beep. Nudge **Machine silence threshold (s)** so the greeting boundary is detected later.

Verify each change with a handful of test calls to real voicemail boxes before rolling it into a campaign.
