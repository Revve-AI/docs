The **Evaluation** tab defines which grading policies apply to one Voice Agent. It doesn't hold a policy of its own — it holds a list of **Scorecards** (a team-wide, reusable object) that you attach to this agent. Once a scorecard is attached, every call the agent completes is graded against it automatically — no one has to sit down and listen to calls to know whether the agent is doing its job.

This is separate from the **General** / **Model Settings** / **Advanced Settings** tabs covered in [Voice Agent Settings](/voice-agents/voice-agent-settings). It's its own tab because it isn't a call-behavior setting — it's a policy that runs _after_ the call, against the finished transcript and tool logs.

<Note>
  Don't confuse this with the **Human Feedback** widget you'll see on an individual call in Call History, which lets a reviewer manually mark that one call as failed and pick an error category. That's a person making an ad-hoc judgment call about a single call. Scorecard evaluation is the opposite: an automatic, policy-driven grading system that scores every call the same way, without a human involved.
</Note>

## Why this is attach-a-scorecard, not configure-a-policy

Earlier versions of this tab let you build a grading policy directly on the agent — metrics, weights, and a pass threshold, configured once and usable only by that one agent. That's gone. The policy now lives on a standalone **Scorecard** object (**Evaluations → Scorecards** in the left nav), and this tab just attaches existing scorecards to this agent.

The reason is that the same grading policy needs to apply in two places that have nothing to do with each other: automatically to this agent's live calls (this tab), and on demand to batches of recordings someone uploads later — a QA sample of last month's calls, or recordings from a legacy dialer being migrated in. Defining "what does a passing call look like" once, as a scorecard, and attaching it wherever it's needed means those two grading paths can't drift apart. Building the policy inline on the agent, the old way, meant the batch-grading side (Scorecards) had to duplicate the whole editor — which is exactly what happened, and why it was replaced.

See [Scorecards](/evaluations/scorecards) for how a scorecard itself is created and edited — this page only covers attaching one to a Voice Agent.

## Attaching a scorecard

Click **Attach Scorecard** (top right, also shown in the empty state) to open the **Attach Scorecards** dialog. It lists your team's scorecards that aren't already attached to this agent, each as a checkbox showing its speaker-scope icon, name, and metric count. Check the ones you want and confirm with **Attach (N)**.

Once attached, scorecards show up in a table:

| Column | Meaning |
| --- | --- |
| **Name** | The scorecard's name, with a speaker-scope icon next to it — a bot icon for a scorecard that scores the **Agent**, a person icon for one that scores the **Caller** (hover either for the tooltip). Click the name to open the scorecard itself. |
| **Description** | The scorecard's own description, if it has one. |
| **Metrics** | How many metrics the scorecard's policy includes. |
| **Action** | A trash icon to detach. |

You can attach up to 20 scorecards to one agent. That cap is generous for normal use, but it's worth noticing if you're approaching it: 20 scorecards means 20 separate grading passes (and, for LLM Judge metrics, 20 sets of judge calls) on every single completed call. If you're attaching that many, check whether some of them are really variations on the same policy that belong in one scorecard with more metrics instead.

### Detaching a scorecard

Click the trash icon on a row to open a confirm dialog titled `Detach "<name>"?`, with the body "This voice agent will no longer be evaluated against this scorecard." Detaching only removes the attachment; the scorecard itself, and any scores it already produced on past calls, aren't affected. It also doesn't retroactively re-grade anything — past calls keep whatever scores they already have.

## Choosing which scorecards to attach

A scorecard grades exactly one side of the conversation — its **speaker scope** (Agent or Caller) is fixed when it's created, and it produces exactly one score. That's a deliberate change from the old model, where a single policy always produced a separate Caller score and Agent score whether you needed both or not. Now you decide:

- **Attach an Agent-scope scorecard** for the case every agent needs: did it follow the script, hit required disclosures, use its tools correctly. This is almost always the first scorecard you attach.
- **Attach a Caller-scope scorecard** when you're tracking something about the caller's side specifically — did they sound satisfied, did they push back on an offer, did the call end in a way that signals a good or bad outcome. Useful for sales and retention agents; often skippable for a purely transactional bot (appointment confirmations, simple lookups) where the caller's tone isn't a signal you act on.
- **Attach both** when you want each graded and tracked independently — for example a compliance scorecard on the agent side and a satisfaction scorecard on the caller side, each with its own pass bar, rather than one blended number that could hide a failure on either side.

Because scorecards are reusable, the same Agent-scope compliance scorecard can be attached to every agent that has to follow that policy — build it once, attach it everywhere it applies.

## Where results show up

Call History shows one badge per attached scorecard on every graded call — its speaker-scope icon, the scorecard's name, its score, and a pass/fail status (green for passed, red for failed, gray while evaluation is still running, was skipped, or hit an error; hover the gray badge for the error detail). Attach three scorecards to an agent, and a graded call shows three separate badges — there's no combined score across scorecards.

Open a call's detail view for the full breakdown: with more than one scorecard, each gets its own tab, and each tab shows that scorecard's score, its pass threshold, and the result of every individual metric that ran.

<Note>
  **Legacy note:** a published version of an agent from before this change, with no scorecards attached to it, may still show a read-only snapshot of its old inline policy on this tab — labeled as the version's legacy evaluation policy. That's historical data for versions published before scorecards existed; it isn't something you can edit, and new versions don't get one.
</Note>

## An example: validating identity before discussing account details

Metrics are generic — nothing here is banking-specific — but they're the kind of building block a bank would use to enforce a hard rule mechanically instead of hoping the agent's prompt is followed. For example, an Agent-scope scorecard's **Tool Called** metric, verifying that the **Built-in Validate** tool (see [Voice Agent Tools](/voice-agents/voice-agent-tools)) ran before the call moved on to discuss account specifics. Because it's a deterministic check, this catches a real process violation on every single call — no LLM judgment involved, no chance of the check itself being fooled by a plausible-sounding transcript.

This is a different mechanism from [Phone Verification](/voice-agents/voice-agent-settings), which confirms a caller's identity by matching a spoken phone number against the one on file for that contact, at call time. Built-in Validate is a general-purpose tool for validating a spoken field — an account number, an ID — against a regex, and a Tool Called metric on a scorecard is one way to confirm the agent actually invoked it, after the fact.

<Note>
  This is one example configuration built from generic metric types, not a claim that Revve ships a banking-specific evaluation feature. Any team can define the same kind of check for whatever field or tool call matters to their process.
</Note>

## The trap: attached with no scorecards, or a scorecard with no metrics

Attaching zero scorecards means the agent's calls simply aren't graded — nothing shows up in Call History. Attaching a scorecard that itself has no metrics produces an empty, meaningless result once it does score a call — there's nothing for it to grade against. Neither failure mode is obvious from this tab alone: check a scorecard's **Metrics** count before relying on the scores it produces here.

For the metric type system itself — Deterministic vs. LLM Judge, the seven deterministic checks, Binary vs. Scale 1-5 scoring, score rubrics — see [Evaluation Metrics](/evaluations/evaluation-metrics). That mechanics is identical whether a metric is attached inside a Scorecard or an Evaluator.
